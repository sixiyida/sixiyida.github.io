MCP 学习笔记

总体架构

  • MCP Hosts: Programs like Claude Desktop, IDEs, or AI tools that want to access data through MCP
  • MCP Clients: Protocol clients that maintain 1:1 connections with servers
  • MCP Servers: Lightweight programs that each expose specific capabilities through the standardized Model Context Protocol
  • Local Data Sources: Your computer’s files, databases, and services that MCP servers can securely access
  • Remote Services: External systems available over the internet (e.g., through APIs) that MCP servers can connect to

这里由图，会话生命周期主要有三个阶段：

  1. 主机初始化客户端，客户端初始化session向服务端发送请求，服务端回复自有能力。
  2. 会话阶段，主要包含三种功能。
  3. 主机向客户端发起终止，客户端向服务端发起中止请求。

对于会话阶段，同样有三种事件循环：

  1. MCP client发起工具/资源请求，server回以响应内容。
  2. 同样的，Server可以向client发起采样请求。
  3. Server有资源更新，可以通知client。

核心概念

Servers offer any of the following features to clients:

  • Resources: Context and data, for the user or the AI model to use
  • Prompts: Templated messages and workflows for users
  • Tools: Functions for the AI model to execute

Clients may offer the following features to servers:

  • Sampling: Server-initiated agentic behaviors and recursive LLM interactions
  • Roots: Server-initiated inquiries into uri or filesystem boundaries to operate in
  • Elicitation: Server-initiated requests for additional information from users

资源（Resources）

MCP支持任意类型的资源，URI按照以下格式来：

1

[protocol]://[host]/[path]
  • file:///home/user/documents/report.pdf
  • postgres://database/customers/schema
  • screen://localhost/display1

具体的请求有：Discovery和Reading，具体请求格式见：https://modelcontextprotocol.io/docs/concepts/resources#typescript

协议

JsonRPC2.0

JSON-RPC 2.0 是一种轻量级、无状态的远程过程调用（RPC）协议，使用 JSON 作为数据交换格式，专为跨语言和跨平台通信设计。

HTTP vs RPC

  • JSON-RPC 2.0

    定位：基于消息的远程过程调用（RPC）协议，关注动作执行（如调用函数deleteUser）。

    设计核心：通过结构化 JSON 消息封装方法名、参数和请求 ID，实现跨语言的服务调用，屏蔽网络细节。

    典型场景：微服务间通信、智能代理指令传递（如 MCP 中的函数调用链）。

  • HTTP

    定位：应用层传输协议，关注资源操作（如DELETE /users/123）。

    设计核心：基于请求-响应模型，利用 URL 定位资源，通过 HTTP 方法（GET/POST/DELETE）操作资源状态。

    典型场景：Web 页面加载、RESTful API 设计（如公开的用户管理接口）。

MCP为何不选用HTTP或gRPC？

需求：1. 双向通信、2. 会话状态维持、3. 松散结构以支持动态发现工具、4. 易于调试。

HTTP不支持1和2。

gRPC不支持3和4。

具体格式可以见：https://modelcontextprotocol.io/docs/concepts/transports

内置传输机制

stdio

使用STDIO在本地标准输入输出流之间传输，这对于本地的进程之间通信非常方便。

Streamable HTTP

Streamable HTTP是可升级为SSE的基于HTTP的传输协议。在这种形态中，MCP Server必须要暴露一个支持GET和POST的端点比如说：

1

https://example.com/mcp

对于Client到Server，所有JSON-RPC格式的message都通过HTTP POST的形式从Client送到Server。

对于Server到Client的响应，可以选择是简单的JSON的响应，或者是建立SSE的流式响应。

对于Server到Client的请求或者是通知，可以使用由Client请求的已经建立的SSE，或者是Client可以主动发起一个空GET请求，此时Server就会把它升级为一个SSE。至此，Server就具备了一个能够主动向Client推送信息的SSE通道。

SSE（已经弃用）

SSE在逻辑上是一个由客户端发起、由服务器同意而建立的从服务器向客户端发消息的单向管道。

以下是一个Server向Client发起的事件：

1
2
3
4

id:11a3eabb-f6fa-4c5a-8d9b-80ffdb040b14
event:endpoint
data:/sse?sessionId=11a3eabb-f6fa-4c5a-8d9b-80ffdb040b14
<空行>

客户端收到每行后，都解析出一个字段，字段名分别为id、event和data（这三个全是SSE规定的标准字段，分别表示事件的ID、类别和数据）。收到空行后，就把前面解析出的字段们放到一起，组合成一个事件，由客户端代码中的事件处理逻辑进行处理。

客户端往服务器发消息时仍然使用传统HTTP请求的方式。客户端发的消息都发往同一个地址，就是此处服务器返回的**/sse?sessionId=11a3eabb-f6fa-4c5a-8d9b-80ffdb040b14**。
