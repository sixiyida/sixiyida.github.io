多线程竞争下的问题

**原子性：**互斥访问，原子操作atomic/串行访问synchronized限制。

**可见性：**直接操作主内存，使用synchronized和volatile实现。

volatile

  1. 读写直达主内存，绕过缓存。
  2. 写以后其他CPU变量的缓存自动失效强制重新加载。

有序性： 指令重排序导致的线程观测其他线程指令执行顺序不一致。

线程的创建方式

  1. 继承Thread类，重写run()方法，用start()方法启动线程。
  2. 实现Runnable接口，函数式传给Thread构造函数。
  3. 实现Callable接口，并将其封装入FutureTask，最后函数式传给Thread构造函数。
  4. Executor框架，executor.submit即可。

如何停止线程运行

  1. 在run方法中判断interrupted状态，如果是的话抛出异常/return。然后直接调用interrupt方法即可。如果线程正在执行Thread.sleep() / Thread.join() / Object.wait()等可中断方法，则解除阻塞并抛出异常。
  2. 根据约定变量显式判断
  3. 通过executor + future取消任务
  4. 有资源的情况下直接销毁资源，会弹出异常以退出

Java线程的状态

NEW：尚未start

RUNNABLE：已经start等待调度/正在运行

BLOCKED：等待锁

WAITING：等待零一线程操作

TIMED_WAITING：具有等待时间的等待

TERMINATED：终止

sleep vs wait

sleep：任意位置调用，和锁无关，超时释放，释放cpu不是放锁

wait：有锁下调用，notify/超时唤醒，作用主要是释放锁

blocked vs waiting

blocked：锁竞争失败进入，到锁可用解开阻塞，自动触发

waiting：特定方法主动唤醒

Java线程安全的保证

synchronized：通过监视器锁实现的同步代码块

volatile：通过直写内存/失效通知机制实现的确保变量在不同线程中的观测一致性

lock/ReentrantLock：更多锁

Atomic：原子操作

ThreadLocal：当前线程中的存储空间

并发安全集合

JUC工具类：如信号量/屏障等

常见锁

监视器锁（sychronized）：

Java提供的原子性内置锁，在同步的代码块前后加上monitorrenter和monitorexit字节码指令，底层依赖操作系统互斥锁。

接着就是可重入锁的逻辑。

1.6后的锁升级机制：

No Lock -> （首次访问同步块）-> Biased Lock(比对线程ID) -> (二线程锁竞争)

-> 轻量级锁 (CAS) -> (CAS失败 )-> 大锁

**轻量级锁：**线程备份锁对象Mark Word，CAS Mark Word指向备份的指针

CAS成功：线程获得锁

CAS失败：检查是否重入，如果重入则进入，如果不是则锁升级

解锁：用CAS还原Mark Word

还原成功：锁释放

还原失败：已经是大锁，唤醒阻塞线程

优化：锁消除（无竞争可能性消除锁）、锁粗化（多锁合一）、自旋锁（减少上下文切换开销）

ReentrantLock

引用计数，进入+1，出去-1，是0则释放锁。

其他锁

读写锁：读写分离的锁，读和写线程分开获取锁

乐观锁：版本号比较

自旋锁/互斥锁：见OS

    • Semaphore：剩余许可数
    • CountDownLatch：未完成计数
2. FIFO 等待队列（CLH 变体）

  • 双向链表结构（非严格的 CLH 队列）

3. 模板方法模式

子类仅需实现关键钩子方法：

  • LockSupport.unpark(thread)：唤醒指定线程

ThreadLocal的原理

每个线程中都有一个独立的ThreadLocalMap，这个东西是哈希表，他的key是ThreadLocal，value是值。

线程池的ThreadLocal泄露问题

由于ThreadLocalMap中的value是强引用，线程不结束GC不能回收，所以由于线程池线程长久运行，导致资源无法自动释放。

CAS的ABA问题

即CS之间存在A->B->A的变化，但是CAS会成功。

解决方法：维护一个版本号（Java的办法）

volatile的作用

  1. 保证变量对所有线程的可见性
  2. 禁止重排序优化：保证写在所有读写完成之后，保证读在所有读写之前进行。

公平锁和非公平锁

**公平锁：**先进等待队列，需要切换两次上下文，慢

非公平锁：先抢锁，失败进等待队列，成功则不需要切换上下文，快

synchronized是非公平锁，可重入锁是公平锁。

线程池

线程池7个参数

核心线程数、最大线程数、阻塞队列、最大淘汰时间、工厂类、淘汰策略、时间单位

拒绝策略

异常拒绝、默拒绝、主线程执行、抛弃最老任务

线程池参数设置

CPU密集型：CPU核+1

IO密集型：CPU核*2

为什么不能用默认executor

FixedThreadPool内部使用无界任务队列，导致如果消费速度小于生产速度，会无限堆积，导致内存占用过多OOM。

CachedThreadPool内部使用无界任务队列，且核心线程数为0，如果生产速度大于消费速度，会导致线程无限创建，导致CPU过载等问题。
