锁

行级锁

**行级锁：**在select for update等场景，即当前读（另一种是快照读MVCC）场景使用。

包括Record Lock, Gap Lock 和 Next-key Lock(前两种的合并)

包括X(Exclusive)和S(share)两种

Gap Lock是只锁相邻两条记录之间的()，NK-lock是锁(]

Gap Lock的X和S型是一样的，都可以重复获取，NK-lock要看右区间的记录锁是否互斥，无限除外。

怎么加？（MySQL8.0.26, 可重复读）

对索引加，基本单位是nk-lock，不同情况可能出现退化为前两种

主键索引等值查询：

记录存在->退化为记录

记录不存在->退化为间隙锁

1

select * from performance_schema.data_locks //查加了什么锁

如果MODE是GAP， LOCK_DATA是右区间界。

主键索引范围查询：

大于：不退化

大于等于：如果等于存在，则左边退化为记录锁，不存在则不退化

小于：最右侧退化为间隙锁

小于等于：若等于存在，则不退化，不存在则最右侧退化为间隙锁

**注：**记录锁属于记录，在GAP锁中是属于LOCK_DATA，右区间界的记录。

二级索引（非唯一）等值查询：

记录不存在：二级索引上GAPLOCK，对于左右端点，能否插入要看二级索引B+树下一条记录有无GAPLOCK

记录不存在的特殊情况：如果是超过了最大id，是next-keyLock

注：二级索引GAPLOCK的LOCK_DATA包含两个值，二级索引和回表的主键索引

记录存在：对二级索引匹配的记录加nk锁，对不匹配的第一个nk锁退化为间隙锁，且在主键索引加记录锁。

对于端点是否能插入，和要看二级索引B+树下一条记录有无LOCK

在不匹配的第一个索引加nk锁的目的：防止幻读（id > lock_id）的情况

二级索引（非唯一）范围查询：不退化，二级的nk和主键的记录锁都加

如果不走索引全表扫描，则所有记录全部加nk锁，全锁，是事故

解决方案：将sql_safe_updates设置为1，此时必须使用where+索引 / limit

当前读的语句：update、delete、select for update，会加意向锁和行锁

死锁的形成

例子：如果两个事务都获取了间隔锁，且希望插入对方间隔，则尝试获取插入意向锁（和间隔锁互斥），环路等待导致死锁。

insert语句加行级锁

**记录之间有间隙锁：**加插入意向锁

注：mysql的锁是先生成锁结构，锁此时是等待状态，再获取锁，如果不能获取则阻塞。

**唯一键（主键或唯一二级索引）冲突：**失败后加S型锁

主键：加S记录锁

唯一二级索引：加S型NK锁

例子：在select for update中，尝试加X型锁，和S型冲突，所以失败。

并发insert导致的唯一键冲突

第一个insert但事务没提交时，构造隐式锁。

第二个insert时，隐式锁变成X型锁，和第二个insert想要获取的S型nk锁冲突

避免死锁的方法

开启主动死锁检测：主动回滚

1

innodb_deadlock_detect = on

undo log

注：对于增删改语句，innodb会隐式启动事务。

特殊处理：delete只在记录上打标记，真正删除由purge线程完成

update非主键列：直接update，且在undolog中记录update之前的值

update主键列：先删再插

undolog的存储形式：由roll_pointer指针形成链表穿起来

bufferpool

指的是innodb引擎中的内存bufferpool。

类似于pagecache，由后台线程实现脏页写回机制。

内存结构：首先申请连续的内存空间，接着按照16kb大小划分出缓存页。

包括数据页、索引页、undo页、插入缓存页、自适应哈希索引、锁信息等。

redolog

**redolog的作用：**对修改实现持久化。

WAL技术(write-ahead logging)：在写入磁盘之前先写入redolog。

每对Bufferpool进行修改就写入redolog，包括undolog的修改

redolog和undolog对比：

redolog是记录修改后，保证持久化

undolog是记录修改前，保证原子化

写入数据和写入redolog对比：

redolog：是顺序写入，高效

数据：是随机写入，低效

redolog也有buffer，落盘时机：

mysql正常关闭，buffer空间超过一半，每一秒写回一次。

innodb_flush_log_at_trx_commit参数：提交事务时的行为

0：不写回，后台线程每隔一秒用write()和fsync()

1：直接持久化到磁盘

2：写入文件（pagecache）由操作系统写回，后台线程每隔一秒用fsync()

redolog存储方式：两个redolog文件循环存储类似于环形队列。

有个tail和head，在tail处写，持久化bufferpool进入数据后更新head。

binlog

Server层的日志， 用于备份恢复、主从复制。三种格式类型：

STATEMENT：记录SQL语句逻辑操作

ROW：记录行数据最终修改情况

MIXED：根据情况使用STATEMENT或者ROW

使用追加写，写满文件就创建新文件继续写，全量日志。

主从复制

主库server层直接写入binlog，后台log dump线程异步将binlog日志发给从库，从库relaylog记录binlog，后台线程异步执行relaylog。

**从库的数量选择：**对主库的资源消耗、网络带宽。

其他模型：

同步模型：要所有从库relaylog记录完毕后返回成功，主库再返回客户端。没法用

异步模型：默认模型，主库宕机就gg

半同步：一部分库返回成功即可。

binlog也有cache，每个线程各一个。

持久化的时机：

binlog_cache_size：超过这个大小就写入

sync_binlog：

0：只write，操作系统控制写回

1：write+fsync

n：write，累积n个以后fsync

注：binlog在语句执行完成后在记录。事务提交时候才持久化。

两阶段提交

问题：如果redolog和binlog一个完成一个不完成，则会出现主从不一致的问题。

内部XA事务：在事务提交后开启，由binlog协调。

将redolog写入拆为prepare和commit，中间插入binlog持久化。

prepare：将内部XID写入redolog并持久化，将状态设置为prepare。

commit：将内部XID写入binlog并持久化，接着将redolog设置为commit。

崩溃时，redolog处于prepare状态，检查binlog中有无XA事务的id，有则提交事务，无则回滚。

**问题：**性能差

磁盘IO次数高

在多事务下，不能保证两者提交顺序一致，需要加锁以保证提交的原子性

上述问题：binlog组提交机制

分为flush(write)、sync(fsync)、commit阶段，每个阶段都有队列，用锁保证事务写入顺序。

redolog的组提交机制

FLUSH链表：管理空闲页，节点是控制块，头结点包括链表头尾地址和控制块数量。

如何管理bufferpool：

传统LRU问题：

**预读失效：**预读时会把相邻的数据页一并加载，为了减少磁盘IO，如果这些没有被访问，且淘汰末尾页，则降低缓存命中率。

**解决方法：**划分LRU的优先级，前面是YOUNG，后面是OLD，预读先加入OLD，真正访问才加入YOUNG区域。

**缓存污染：**当扫描大量数据，会淘汰大量热数据，导致命中率下降。

**解决方法：**提高加入YOUNG的门槛，记录第一次OLD被访问的时间，如果后续访问时间超过第一次1s，则放入young区域。

**Linux的做法：**第二次的时候将才升级到active，比MYSQL简单。
